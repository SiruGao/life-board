# 公开发布与 GitHub Star 增长计划

## 原则

Socratic Kernel 不购买 Star、不参与 Star 互换、不使用机器人账号，也不通过夸大功能制造传播。

Star 的正确来源是：

1. 用户在几秒内理解项目；
2. 在几分钟内成功体验或部署；
3. 认为项目代表一个值得继续的方向；
4. 愿意再次回来、贡献或把它分享给真正相关的人。

## 北极星指标

Star 是公开关注指标，不是产品价值本身。

发布期同时观察：

- README 到在线体验的点击率；
- 首次审议开始率和完成率；
- 七天内第二次审议率；
- Vercel 一键部署点击率；
- Issue、Discussion、Fork 和外部贡献数量；
- 判断摘要导出和隐私分享使用率；
- 用户是否能说明产品与普通聊天助手的区别。

## 发布前门槛

## 发布叙事

不要把项目描述为“又一个 AI 聊天应用”。

核心叙事：

> AI 已经让答案变得廉价。Socratic Kernel 试图保护的是提出问题、选择标准和承担结论的能力。

三个可验证差异：

1. 用户必须在看到 AI 追问前写下自己的立场；
2. 系统默认不替用户形成最终结论；
3. 完成后得到的是用户亲自确认的判断档案，而不是一段 AI 建议。

## 30 天发布节奏

### 第 1 周：封闭测试

目标：确认产品能独立使用，而不是确认朋友是否喜欢设计。

- 邀请 20–30 名目标用户；
- 每人独立完成一次审议；
- 记录开始、完成、卡住和再次使用；
- 完成 8–10 次访谈；
- 修复最常见的三个阻塞点；
- 选择 3 个可匿名公开的使用案例。

### 第 2 周：仓库与内容准备

- 更新真实界面截图和 GIF；
- 发布 v0.6 Release；
- 写一篇方法论文章；
- 准备中英文演示视频；
- 开启 Discussions，并发布 Welcome、Ideas、Show and tell 三个讨论；
- 确认所有公开内容不包含用户私密审议正文。

### 第 3 周：开发者渠道发布

分批发布，不在同一天复制同一段广告：

- GitHub Release；
- Hacker News：重点说明问题、设计取舍和开源架构；
- Reddit：选择 critical thinking、self-hosted、local-first、AI literacy 等相关社区；
- X / LinkedIn：演示一个完整审议闭环；
- Product Hunt：突出在线体验、一键部署和隐私架构。

每个渠道根据其用户重写标题和正文，不做跨社区垃圾投放。

### 第 4 周：中文渠道与复盘

- 知乎：AI 是否正在削弱人的判断主体性；
- 少数派：本地优先的思考工具与设计过程；
- B 站：3–5 分钟产品演示；
- 小红书：视觉设计、使用场景和判断卡；
- 即刻：开发日志和公开问题。

复盘：

- 哪个渠道带来真实完成用户，而不只是访问；
- 哪个使用案例最容易被理解；
- 哪个功能促成第二次使用；
- 哪种内容带来贡献者，而不只是 Star。

## Star 转化设计

可以在以下位置提出 Star 请求：

- README 完成价值说明之后；
- 用户成功完成一次审议并主动打开分享区之后；
- 发布视频结尾；
- Release notes 末尾。

不要：

- 首次打开应用立即弹窗索要 Star；
- 用功能解锁交换 Star；
- 声称 Star 等于研究认可；
- 高频重复提醒。

建议文案：

> 如果你认为“AI 时代仍需要保护人的判断自主权”是一个值得继续的方向，可以 Star 仓库，让更多愿意参与这场讨论的人看到它。

## 社区结构

### Issues

用于：

- 可复现缺陷；
- 已形成范围的功能任务；
- 文档、翻译和无障碍任务。

### Discussions

用于：

- 产品方向；
- 哲学与方法争议；
- 使用案例；
- Show and tell；
- 尚未形成实现方案的想法。

### Pull Requests

必须说明：

- 对用户主体性的影响；
- 数据和隐私变化；
- 测试结果；
- 已知限制。

## 首批目标

不要以“十万 Star”为短期目标。

阶段目标：

- **0 → 100**：证明定位能够被第一批开发者和思考工具用户理解；
- **100 → 1,000**：证明产品能够真实运行、持续更新并形成外部贡献；
- **1,000 → 10,000**：建立扩展、桌面端、多语言、研究评估和稳定社区；
- **10,000+**：取决于产品是否形成独特类别，而不是发布技巧。

## 禁止的增长方式

- 购买或交换 Star；
- 机器人账号或批量私信；
- 向无关仓库提交宣传 Issue；
- 冒充用户评价；
- 隐藏遥测；
- 把用户私密判断用于营销素材；
- 用焦虑、羞耻或连续签到制造依赖。
