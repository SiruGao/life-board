# Socratic Kernel 增长与订阅路线

## 目标

Socratic Kernel 的增长目标不是提高依赖、延长对话时长或制造焦虑，而是让更多用户完成一次高质量审议，并在真实决策、阅读和 AI 使用场景中反复受益。

增长必须同时满足三项约束：

1. 不通过虚假 Star、刷量、垃圾推广或暗黑模式获取数字；
2. 不收集与产品价值无关的敏感数据；
3. 不把“用户离不开产品”作为成功指标。

## 北极星指标

首要指标：**每周完成至少一次完整审议的活跃用户数**。

配套指标：

- 首次访问到开始审议的转化率；
- 开始审议到完成审议的转化率；
- 7 日内第二次审议率；
- 4 周留存率；
- 用户主动导出、分享或安装 PWA 的比例；
- 用户是否能在不依赖系统时提出反例、说明证据和形成自己的结论。

GitHub Stars、邮件订阅数和社交媒体曝光只作为分发指标，不代替真实使用和留存。

## 用户增长漏斗

```text
看到项目
→ 理解它解决什么问题
→ 30 秒内进入示例
→ 5 分钟内完成第一次审议
→ 得到一份可保存的判断摘要
→ 在下一次真实场景中再次使用
→ 订阅更新或升级高级能力
→ 主动推荐给有相同问题的人
```

## 第一阶段：激活（Activation）

目标：让新用户不需要阅读长文档，也能在 30 秒内开始并在 5 分钟内得到明确价值。

必须完成：

- 首次访问引导：用三屏解释“先写立场、接受追问、形成自己的判断”；
- 一键示例：预置“是否应该使用 AI 完成重要作业”“是否接受一份工作”等示例；
- 首次审议进度提示和预计步骤，不显示刺激性倒计时；
- 完成页生成结构化摘要：原始立场、最强证据、最强反方、最终判断、下一步行动；
- 完成后提供三个自然动作：保存、导出、开始另一次审议；
- README 首屏增加真实操作 GIF，而不只展示静态概念图。

阶段门槛：至少邀请 30 名目标用户测试，其中 15 人独立完成一次审议，10 人在 7 日内再次使用。

## 第二阶段：留存（Retention）

目标：让用户因为真实需求回来，而不是因为签到、连续天数或通知压力回来。

核心回访场景：

- 每周判断复盘：回看本周做过的判断及后来发生的事实；
- AI 使用前审计：在把任务交给 AI 之前确认外包边界；
- 阅读质疑：从网页选中文字后直接启动分析；
- 决策模板：工作、学习、消费、关系、项目优先级；
- “我改变了什么”：记录新证据如何改变确信程度。

不使用强制签到、损失厌恶倒计时、连续使用惩罚或无关推送。

阶段门槛：7 日第二次审议率达到 25%，4 周留存率达到 15% 后，再扩大推广。

## 第三阶段：订阅产品

开源本地核心继续免费。付费订阅必须提供额外基础设施和持续服务，而不是人为阉割基本思考能力。

### Free

- 本地优先的五种审议模式；
- 本地历史记录与导入导出；
- PWA 和基础浏览器扩展；
- 社区模板与公开协议。

### Plus（价格实验：¥29–39/月）

- 端到端加密的多设备同步；
- 高级审议模板与自定义模板；
- 周度复盘报告；
- 浏览器扩展中的跨网页工作流；
- 优先体验新功能。

### Pro（价格实验：¥59–89/月）

- 可替换的模型追问层；
- 反迎合审查器与来源核验；
- 可编辑的个人认知模式图谱；
- 更完整的长期趋势和导出报告；
- 自带模型密钥（BYOK）与本地模型支持。

### Team / Education

- 团队或课程模板；
- 匿名化、聚合式使用洞察；
- 管理员不能读取个人私密审议正文；
- 部署、培训与支持服务。

正式定价前先使用付费意向页和访谈验证，不在尚未形成留存前投入复杂支付系统。

## 第四阶段：分发

### GitHub

- README 首屏：一句话价值、真实 GIF、在线体验、安装入口；
- 固定发布节奏：每 2–4 周一个可感知版本；
- 每次 Release 附带截图、变化说明和下一步；
- 建立 Good First Issue、Discussion、贡献者名单和公开路线图；
- 通过真实使用案例获得 Star，不购买或交换 Star。

### 内容渠道

每项功能都转化为一个可传播的问题，而不是只发布开发日志：

- “AI 给出的答案很完整，为什么我仍然不该直接相信？”
- “如何判断自己正在把价值判断外包给 AI？”
- “做重要决定前，最值得写下的七个问题。”
- “一个反对你的人，怎样提出最强版本的反驳？”

内容形式：短视频演示、图文案例、真实匿名审议、开发日志、方法论文章和公开评测。

优先渠道：GitHub、Hacker News、Product Hunt、Reddit、X；中文渠道优先知乎、少数派、即刻、B 站和小红书。

### 推荐机制

只在用户完成并明确认可一次审议后出现分享入口。分享内容默认不包含私密正文，只生成：

- 产品链接；
- 用户主动选择公开的模板；
- 去身份化的审议方法卡片；
- 可选推荐码和候补名单位置。

## 90 天执行顺序

### 第 1–2 周：首次体验

- 完成首次访问引导和一键示例；
- 改造完成页与导出摘要；
- 制作真实 GIF 和三张产品截图；
- 建立基础反馈入口。

### 第 3–4 周：真实用户验证

- 邀请 30–50 名目标用户；
- 记录开始率、完成率和第二次使用率；
- 进行至少 10 次访谈；
- 删除无人使用的功能，修复主要阻塞点。

### 第 5–8 周：留存能力

- 开发浏览器扩展最小版本；
- 增加每周复盘和判断变化记录；
- 发布 10 个高质量模板；
- 建立可选邮件更新订阅。

### 第 9–12 周：公开发布

- 发布 v0.3；
- 同步发布 GitHub Release、演示视频和方法论文章；
- 分批提交到 Product Hunt、Hacker News 和垂直社区；
- 开启 Plus 付费意向测试，而不是立即承诺完整商业版。

## 首批实验

| 实验 | 假设 | 主要指标 | 判定方式 |
| --- | --- | --- | --- |
| 一键示例 | 空白输入是首次流失主因 | 首次开始率 | 与无示例版本比较 |
| 五分钟审议 | 六到七题对新用户过重 | 完成率、回答质量 | 比较 4 题和 6 题流程 |
| 判断摘要 | 可保存结果会提高价值感 | 导出率、再次使用率 | 完成页 A/B 测试 |
| 周度复盘 | 真实复盘比签到更能带来留存 | 4 周留存率 | 小范围邀请测试 |
| 付费意向页 | 用户愿意为同步、扩展和模型层付费 | 点击率、访谈报名率 | 不提前收费，先验证意向 |

## 当前优先级

1. 首次访问引导与示例；
2. 完成页和可分享摘要；
3. 真实产品 GIF；
4. 反馈与可选更新订阅；
5. 浏览器扩展 MVP；
6. 周度复盘；
7. 付费意向验证；
8. 支付与云同步。

在前四项完成并获得真实留存数据之前，不优先开发复杂账号、支付或团队后台。
